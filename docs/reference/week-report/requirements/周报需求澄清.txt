基于我下面的需求澄清，新输出一个新的markdown格式的需求文档V2.0版本，包括我下面的需求澄清内容细化。  

1. 交付范围

  - 本次是否只交付 Requirement 文档、Design 文档和 Demo 原型，不交付真实前后端源码？（是的，但是Demo原型是非抛弃原型，是后续连接了数据库后也真实复用的原型）
  - Demo 的“完全可运行、可交互”标准是什么：仅页面跳转和表单交互，还是要包含本地模拟数据增删改查与状态流转？（需要模拟数据，但是不需要连接数据库）
  - Demo 是否允许纯前端静态实现，数据仅做模拟，不接数据库？（是的，不连接数据库）

  2. 一级菜单范围

  - 请确认一级菜单最终清单。
  - 是否包含：首页、待我填报、待我审核、我填报的周报、周报查看、周报模板管理、基础配置、个人信息、系统公告？（不需要系统公告一级菜单）
  - “系统公告”是首页模块，还是还需要独立公告管理菜单？（不需要系统公告一级菜单）

  3. 用户与组织规则

  - 用户唯一标识以什么为准：工号、登录名，还是两者都唯一？（工号就是登录用户名，不存在单独的用户名）
  - 用户状态有哪些：启用/停用，还是还包括锁定、离职？（只有启用停用）
  - 部门字段需要哪些：部门编码、部门名称、排序、状态，还是只要部门名称？（需要部门编码，部门名称，状态）

  4. 角色与权限规则

  - 是否有固定预置角色？如果有，请给出角色清单。（预留超级管理员角色接口，可以访问操作所有功能）
  - 是否有默认超级管理员角色？（预留超级管理员角色接口，可以访问操作所有功能）
  - 菜单权限是否只做到菜单级，不做按钮级、数据级权限？（是的）
  - 周报“查看角色”能看到该模板下所有人的周报，还是只看特定部门/范围？（特定部门范围的）

  5. 审核规则

  - 一个用户是否只能维护一个直属领导？（是的）
  - 如果没有直属领导，提交时如何处理？（必须又直属领导）
  - 审核人是以“提交当时的直属领导”为准，还是“当前最新直属领导”为准？（提交当时的直属领导）
  - 领导自己的周报由谁审核？（不考虑这个问题，最高领导不用填周报，因此所有填周报的人都有直属领导）
  - 审核后是否允许退回重填？如果允许，需要哪些状态：待审核、已退回、已审核？（允许退回，状态就是待审核、已退回、已审核，还包括暂存的草稿状态）

  6. 周报周期与填报时间

  - 周报周期如何定义：自然周一到周日，还是其他周期？（周一到周日）
  - “开始时间、结束时间”表示周报覆盖周期，还是填报窗口？（周报覆盖周期）
  - 周六、周日之外，是完全不能编辑，还是不能提交但可以暂存？（完全不能编辑）
  - 是否允许补报历史周报？（不允许）
  - 每周任务何时生成？是否按“用户 + 模板 + 周期”自动生成一条任务？（不需要，用户进入手工增加当周周报）

  7. 模板结构

  - 一个“数据对象”是单条记录，还是可录入多行的表格对象？（可录入多行的表格对象）
  - 是否允许在一个对象中新增/删除多行，例如多个项目、多条事项？（允许）
  - 数据对象是否需要：名称、编码、顺序、说明、是否必填？（是的，需求）
  - 数据项是否需要：名称、编码、类型、控件、顺序、是否必填、默认值、长度/精度？（是的，需要）

  8. 字段类型与控件范围

  - 当前是否只支持三种基础类型：字符、数字、日期？（是的）
  - 字符控件是否只支持：单行文本、多行文本、下拉单选？（是的）
  - 是否明确不需要：单选、多选、附件、富文本、日期时间、人员选择等控件？（不需要）
  - 数字类型是否区分整数和小数？（不区分）

  9. 提报与复制规则

  - “编制时间”和“报告时间”分别是什么意思？是否自动生成，是否可编辑？（编制时间就是提交周报的时间，报告时间是周一到周日的周期时间段）
  - “还原上周周报”复制的是上周已提交版本，还是最后暂存版本？（上周已经提交版本）
  - 如果上周模板结构变化，复制时如何处理：按同名字段迁移，还是不支持复制？（不支持复制）

  10. 查看与审核页面规则

  - 审核意见是否必填？（不是）
  - 审核是否只允许一次，还是允许保留多次审核历史？（只允许一次）
  - “周报查看”功能可查看哪些状态：待审核、已审核、全部已提交？（已审核）

  11. 首页统计口径

  - 首页统计是按当前登录人统计，还是管理员看全局统计？（当前登录人统计）
  - “待填任务、已填任务、总任务、完成率”分别按什么口径计算？
  - 柱状图统计的是“应填 vs 已填”，还是“应填 vs 已审核”？（应填 vs 已审核）
  - 是否展示所有部门，包括值为 0 的部门？（包括所有部门）

  12. 数据库规范

  - 主键统一用自增整数还是 UUID？（都可以）
  - Flag 字段具体含义是什么：逻辑删除标志，还是通用状态位？（逻辑删除标记）
  - 是否需要统一定义状态枚举值和字典编码规则？（需要）
  - 是否需要预置演示数据、演示账号、演示角色？（需要演示账号，默认超级管理员角色）

  可后补
  这些点不影响先产出主需求框架，但会影响细节质量和 Demo 完整度。

  1. 登录与安全

  - 是否需要首次登录强制改密？
  - 密码复杂度是否有要求？
  - 是否需要输错次数锁定？
  - 是否需要找回密码能力？

  2. 列表交互

  - 是否统一支持分页、排序、关键字查询？
  - 是否需要导出 Excel/PDF？
  - 是否需要批量审核、批量查看、批量删除？

  3. 模板生命周期

  - 模板是否需要发布、停用、复制、版本管理？
  - 模板修改后，是否影响历史已填周报？

  4. 公告管理

  - 公告需要哪些字段：标题、内容、发布人、发布时间、有效期、置顶、状态？
  - 公告是否区分可见范围：全员、按角色、按部门？

  5. 界面细节

  - 是否严格沿用 Demo-Ref 的布局、配色、导航风格？
  - 未提供参考页的模块，是否允许我按同一设计语言自行补全？

  6. 文档组织

  - Requirement 下每个一级菜单生成一份需求文档，命名规则是否需要你指定？
  - 数据库设计文档是否需要 ER 图、索引设计、状态流转图？
