# 周报管理系统需求文档 V2.0

## 1. 文档说明

- 文档名称：周报管理系统需求文档 V2.0
- 文档目的：用于指导周报管理系统 Demo 原型、后续数据库设计文档以及后续可复用系统实现
- 编写依据：`profile.md` 及本轮需求澄清结果
- 文档范围：本版本为总需求文档，后续可按一级菜单拆分为独立需求文档

## 2. 项目目标

建设一个用于公司员工填写、提交、查看周报，并由直属领导审核的管理系统。系统支持基础配置、周报模板配置、周报填报、周报审核、周报查看和首页统计分析。

本项目当前阶段的目标不是交付真实业务源码，而是交付一套可运行、可交互、可复用的 Demo 原型及配套需求与设计文档，为后续连接数据库和继续开发奠定基础。

## 3. 交付范围

### 3.1 本阶段交付物

- `Requirement` 目录下的需求文档
- `Design` 目录下的数据库设计文档
- `Demo` 目录下的可运行、可交互原型程序

### 3.2 本阶段不交付内容

- 不交付真实后端源码
- 不交付真实数据库访问逻辑
- 不交付生产可部署版本

### 3.3 Demo 原型要求

- Demo 为非抛弃型原型，后续连接数据库后应尽量直接复用
- Demo 允许采用纯前端静态实现
- Demo 中必须具备模拟数据
- Demo 中必须支持页面跳转、表单交互、列表查询、状态变化、模拟增删改查
- Demo 不连接真实数据库
- Demo 风格严格参考 `Demo-Ref` 子目录中的示例页面

## 4. 技术与实现约束

### 4.1 技术方向

- 后续正式系统技术路线预定为 `FastAPI + React + TypeScript`
- 当前阶段仅产出需求文档、数据库设计文档和前端可运行 Demo

### 4.2 数据库约束

- 后续数据库采用 `SQLite`
- 表名、字段名使用大写英文字母，多个单词使用下划线分隔
- 所有业务表统一保留以下审计字段：
  - `CREATE_BY`
  - `CREATE_TIME`
  - `UPDATE_BY`
  - `UPDATE_TIME`
  - `FLAG`
- `FLAG` 表示逻辑删除标记
- 主键可采用自增整数或 UUID，数据库设计阶段统一确定即可
- 需要统一定义状态枚举和数据字典编码规则

## 5. 用户角色与权限模型

### 5.1 角色定义

- 超级管理员：预留接口，可访问和操作系统所有功能
- 普通业务角色：按配置参与周报填报、查看等业务

### 5.2 权限控制原则

- 权限控制仅到菜单级别
- 不做按钮级权限控制
- 不做数据级精细权限控制
- 菜单可绑定多个角色
- 一个用户可配置多个角色
- 超级管理员默认拥有全部菜单访问能力

### 5.3 周报业务角色

- 填报角色：具备某个模板填报权限的角色
- 查看角色：具备某个模板查看权限的角色
- 审核角色不单独配置，审核人由“提交时的直属领导”自动确定

## 6. 用户与组织规则

### 6.1 用户规则

- 工号即登录账号
- 不存在单独的用户名字段作为登录标识
- 用户信息至少包括：
  - 工号
  - 登录密码
  - 中文姓名
  - 所属部门
  - 直属领导
  - 用户状态
  - 角色信息
- 用户状态仅包括：
  - 启用
  - 停用
- 密码必须加密存储
- 每个用户只能维护一个直属领导
- 所有允许填报周报的用户必须配置直属领导

### 6.2 部门规则

- 不考虑多级部门结构
- 部门信息至少包括：
  - 部门编码
  - 部门名称
  - 状态

## 7. 左侧菜单结构

左侧菜单采用 `outlook bar` 风格，仅保留两个一级菜单组：

1. 周报填报和查看
2. 基础配置

其中二级页面结构如下：

### 7.1 周报填报和查看

- 首页
- 待我填报
- 待我审核
- 我填报的周报
- 周报查看

### 7.2 基础配置

- 周报模板管理
- 用户信息维护
- 部门信息维护
- 数据字典维护
- 角色维护
- 菜单维护
- 个人信息

说明：

- 不设置“系统公告”一级菜单
- 系统公告只作为首页展示模块存在

## 8. 业务总流程

### 8.1 主流程

1. 超级管理员或管理员维护部门、用户、角色、菜单、数据字典
2. 管理员创建周报模板，并为模板配置填报角色和查看角色
3. 具备填报权限的用户在允许时间内手工新增当周周报
4. 用户填写周报内容，可暂存草稿
5. 用户提交周报后，系统按提交时的直属领导生成待审核数据
6. 直属领导进入待我审核页面查看并审核
7. 审核通过后状态变更为已审核
8. 审核退回后状态变更为已退回，填报人可在允许时间内重新编辑并再次提交
9. 具备查看角色的用户可查看本部门范围内已审核周报

### 8.2 时间规则

- 周报周期为自然周，即周一至周日
- 周报的开始时间和结束时间表示周报覆盖周期
- 进入待我填报页面后，只要当前用户具备某个模板的填报权限，就必须显示一行待填报数据
- 周一至周五允许填写并暂存草稿
- 仅允许在周六和周日进行正式提交
- 不允许补报历史周报
- 不自动预生成周报任务，由用户进入系统后手工新增当周周报

## 9. 状态定义

周报状态统一如下：

- 草稿：用户已暂存但未提交
- 待审核：用户已提交，等待直属领导审核
- 已退回：直属领导审核退回，待用户重新修改提交
- 已审核：直属领导已审核通过

状态规则说明：

- 草稿状态允许修改
- 已退回状态允许继续编辑，且仅可在周六、周日再次正式提交
- 待审核状态不允许填报人修改
- 已审核状态不允许再次修改
- 审核仅允许一次，不保留多轮审核记录

### 10.1 用户信息维护

支持维护用户基础信息、角色信息和直属领导信息。

功能要求：

- 编辑用户

- 启用/停用用户

字段要求：

- 密码
- 中文姓名
- 所属部门
- 直属领导
- 用户状态

业务规则：

- 工号唯一
- 工号即登录账号

### 10.2 部门信息维护

支持维护部门信息。

字段要求：

- 部门编码
- 部门名称
- 状态

业务规则：

### 10.3 数据字典维护

用于维护系统中可配置的基础字典数据。

典型字典示例：

- 项目类型

字段建议：

- 字典类型编码
- 字典类型名称
- 字典项编码
- 字典项名称
- 排序号
- 状态

### 10.4 角色维护

支持维护角色及角色下用户信息。

功能要求：

- 新增角色
- 编辑角色
- 查看角色下用户
- 为角色分配用户

### 10.5 菜单维护

支持维护系统菜单及菜单访问角色。

功能要求：

- 支持菜单层级结构
- 维护菜单名称、菜单路径、上级菜单、排序
- 为菜单配置可访问角色

业务规则：

### 10.6 个人信息

登录用户可查看自己的个人信息，并修改密码。

功能要求：

- 查看工号、姓名、部门、角色等个人信息
- 修改登录密码

## 11. 周报模板管理模块需求

### 11.1 模板管理目标

周报模板是系统核心能力。管理员可灵活定义模板、数据对象、数据项和填报/查看角色，使不同岗位人员按模板进行周报填报、审核和查看。

### 11.2 模板基础信息

每个周报模板至少包括：

- 模板名称
- 模板编码
- 模板说明
- 状态
- 填报角色
- 查看角色
- 查看部门范围

说明：

- 查看角色看到的数据范围是特定部门范围，不是全量数据
- 具体部门范围需要在模板配置中明确
- Demo 原型中“填报角色”“查看角色”“查看部门范围”均使用“文本框 + 按钮挑选”的方式维护
- 点击按钮后弹出独立选择窗口，支持多选，选择结果以逗号形式回填显示

### 11.3 模板结构

一个周报模板可包含多个数据对象。

每个数据对象可包含多个数据项，并以可录入多行的表格对象形式呈现。

数据对象配置项包括：

- 对象名称
- 对象编码
- 显示顺序
- 对象说明
- 是否必填

数据项配置项包括：

- 数据项名称
- 数据项编码
- 数据类型
- 控件类型
- 显示顺序
- 是否必填
- 默认值
- 长度或精度

### 11.4 数据类型与控件范围

数据类型只支持：

- 字符
- 数字
- 日期

控件规则如下：

- 字符类型支持：
  - 单行文本框
  - 多行文本框
  - 下拉单选框
- 数字类型不区分整数和小数
- 日期类型只支持日期，不支持日期时间
- 下拉单选框的数据来源必须配置为某个数据字典项

明确不支持的控件类型：

- 单选按钮组
- 多选框
- 附件上传
- 富文本
- 日期时间
- 人员选择器

### 11.5 多行录入规则

- 每个数据对象支持新增行
- 每个数据对象支持删除行
- 不同数据对象彼此独立
- 同一模板下各对象按配置顺序展示

## 12. 待我填报模块需求

### 12.1 功能定位

用于填报人查看自己可填写的周报并手工新增本周周报。

### 12.2 列表查询条件

列表至少支持以下查询条件：

- 开始时间
- 结束时间
- 周报名称

### 12.3 列表内容

列表建议展示以下字段：

- 周报名称
- 周报周期开始时间
- 周报周期结束时间
- 当前状态
- 编制时间
- 操作

### 12.4 新增与编辑规则

- 用户手工新增当周周报
- 仅可新增当前自然周周报
- 不允许新增历史周报
- 进入页面后，对当前用户具备填报角色的模板均需展示待填报行
- 周一至周五允许进入填报页面并暂存草稿
- 周六、周日允许正式提交
- 草稿和已退回状态允许继续编辑
- 待审核和已审核状态不允许编辑

### 12.5 填报页面组成

填报页面由标题区和正文区组成。

标题区字段包括：

- 姓名
- 部门
- 编制时间
- 报告时间
- 开始时间
- 结束时间

字段规则：

- 编制时间为用户提交周报的时间
- 报告时间为当前周一至周日的周期范围
- 开始时间和结束时间为周报覆盖周期
- 姓名和部门自动带出

正文区规则：

- 自动根据周报模板加载所有数据对象
- 每个数据对象以独立表格区域显示
- 每个对象支持按模板定义渲染字段和控件
- 页面初始化时加载默认行或空行

### 12.6 还原上周周报

- 支持“还原上周周报”功能
- 复制来源为上周已提交版本
- 如果上周周报模板结构发生变化，则不支持复制
- 不允许从草稿版本复制
- 不允许从未提交版本复制

### 12.7 暂存与提交

- 用户可暂存为草稿
- 用户可提交周报
- 提交时必须校验：
  - 当前是否为周六或周日
  - 是否已配置直属领导
  - 必填项是否填写完整
- 提交成功后：
  - 状态变为待审核
  - 审核人固化为提交当时的直属领导
  - 填报人不可再修改

补充说明：

- 暂存草稿不要求直属领导和必填项校验通过
- 正式提交才执行完整校验

## 13. 待我审核模块需求

### 13.1 功能定位

直属领导查看待自己审核的周报，并完成审核通过或退回。

### 13.2 列表规则

列表展示当前登录用户作为审核人的待审核周报。

建议展示字段：

- 周报名称
- 填报人
- 填报部门
- 周报周期
- 提交时间
- 当前状态
- 操作

### 13.3 审核页面规则

- 页面以只读方式展示周报内容
- 不再显示可编辑控件
- 以表格化网页方式展示填报结果
- 支持填写审核意见
- 审核意见非必填

### 13.4 审核动作

支持以下动作：

- 审核通过
- 审核退回

业务规则：

- 审核只允许一次
- 审核通过后状态变为已审核
- 审核退回后状态变为已退回
- 已审核或已退回数据不允许再次审核

## 14. 我填报的周报模块需求

### 14.1 功能定位

填报人查看自己历史填报的周报及审核结果。

### 14.2 列表展示字段

建议至少展示：

- 周报名称
- 周报周期
- 当前状态
- 编制时间
- 审核人
- 审核意见
- 操作

### 14.3 页面能力

- 可查询本人所有周报
- 可查看周报详情
- 可直接在列表中查看审核意见
- 对于草稿或已退回状态，可跳转继续编辑
- 对于待审核和已审核状态，仅可查看

## 15. 周报查看模块需求

### 15.1 功能定位

具备查看角色的用户查看有权限范围内的已审核周报。

### 15.2 数据范围规则

- 仅可查看已审核状态周报
- 仅可查看模板已配置查看角色的数据
- 仅可查看模板配置范围内的特定部门数据
- 不允许在该模块执行审核
- 不允许在该模块修改周报

### 15.3 列表展示字段

建议至少展示：

- 周报名称
- 填报人
- 所属部门
- 周报周期
- 审核时间
- 操作

## 16. 首页模块需求

### 16.1 页面布局

首页采用以下布局结构：

- 顶部 `top` 栏
- 左侧 `outlook bar` 导航菜单
- 中间功能区内容页面
- 中间功能区支持多标签页 `tab` 打开，标签可手工关闭
- 待我填报进入填报页、周报查看进入详情页、待我审核进入审核页时，均在右侧新标签页中打开，不使用弹窗

### 16.2 展示内容

首页展示以下内容：

- 待填任务
- 已填任务
- 总任务
- 完成率
- 系统公告
- 本周分部门统计图表

### 16.3 统计口径

首页统计按当前登录用户口径展示。

V2.0 默认统计口径如下：

- 总任务：当前登录用户在本自然周有权填报的模板数量
- 已填任务：当前登录用户在本自然周已提交的周报数量，状态包括待审核和已审核
- 待填任务：总任务减已填任务
- 完成率：已填任务除以总任务

说明：

- 草稿状态不计入已填任务
- 已退回状态不计入已填任务

### 16.4 图表口径

- 图表类型为柱状图
- 统计维度为部门
- 展示口径为“本周应填总数 vs 本周已审核总数”
- 需展示所有部门，包括值为 0 的部门

### 16.5 系统公告

- 系统公告仅作为首页展示区块
- 当前阶段可采用模拟数据展示
- 本文档不要求建设独立公告管理菜单

Demo 中需要预置模拟数据，至少包括：

- 演示账号

- 角色数据
- 菜单数据
- 数据字典数据
- 周报模板数据

- 首页统计数据

## 18. 非功能要求

### 18.1 易用性

- 页面风格简洁、美观、易用
- 交互逻辑清晰
- 表格、表单、按钮、状态标识清楚明确

### 18.2 一致性

- 所有页面视觉风格需与 `Demo-Ref` 保持一致
- 同类列表页、表单页、详情页的交互方式保持一致

### 18.3 可复用性

- Demo 页面结构、组件结构、交互结构应为后续真实开发复用做准备
- 后续接入数据库时，应尽量少改页面结构和交互流程

## 19. 版本结论

本版需求在原始需求基础上，已补充明确以下关键规则：

- Demo 为可复用的非抛弃型原型
- 系统采用 Outlook Bar 风格左侧导航，仅保留“周报填报和查看”“基础配置”两个一级菜单组
- 系统不设置系统公告一级菜单
- 工号即登录账号
- 直属领导为唯一且必填
- 审核人按提交时直属领导固化
- 周报允许退回重填
- 待我填报页面中，具备模板填报权限的用户必须看到对应待填报行
- 周报支持周中填写并暂存草稿
- 周报仅允许周六、周日正式提交
- 周报不允许补报历史周
- 周报由用户手工新增当周数据
- 模板对象支持多行表格录入
- 查看角色仅看特定部门范围内已审核数据
- 首页统计采用当前登录用户口径

本需求文档可作为后续拆分一级菜单需求文档、输出数据库设计文档和开发 Demo 原型的基线版本。
