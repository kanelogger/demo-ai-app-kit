# 基础管理系统概述

该系统用于快速搭建后台管理类 Web 应用，为后续业务系统提供账号认证、权限控制、组织人员、菜单路由、数据字典、文件附件、通知消息、日志审计、系统配置等通用基础能力，适合作为各类 PC 端后台项目的基础模板。

# 基础配置功能说明

-- 需要有用户信息维护功能，维护系统用户的基础信息。用户信息包括用户工号、登录名、密码、中文姓名、手机号、邮箱、所属部门、岗位、用户状态等。密码需要加密存储。

-- 需要支持用户角色维护。一个用户可以拥有多个角色，角色用于控制该用户可以访问的菜单和功能范围。

-- 需要有部门信息维护功能。部门信息用于组织人员归属，可以先支持单层级部门结构，后续根据业务需要扩展为多级结构。

-- 需要有岗位信息维护功能。岗位用于描述人员在组织中的职责，如管理员、运营人员、审核人员、普通用户等。

-- 需要有角色维护功能。角色信息包括角色名称、角色编码、角色状态、角色说明等。角色维护界面需要支持维护该角色下的用户信息。

-- 需要有菜单维护功能。菜单需要支持层级结构，在菜单维护处需要维护菜单名称、菜单编码、菜单图标、菜单排序、菜单路由地址、组件路径、显示状态等信息。

-- 需要有菜单授权功能。一个菜单可以配置多个角色可访问，系统登录后根据当前用户角色动态生成可访问菜单。

-- 权限控制第一阶段默认控制到菜单级别。按钮级权限、接口级权限可以作为扩展能力预留，但不作为基础模板第一阶段的强制能力。

-- 需要有数据字典维护功能。系统中常用的状态、类型、来源、优先级、标签、开关值等枚举信息统一在数据字典中维护。

-- 需要有个人信息查看功能。登录后的用户可以查看自己的个人信息，并支持修改密码。

# 账号认证和登录说明

1. 系统需要提供登录功能，用户输入登录名和密码后完成身份认证。
2. 登录成功后系统返回访问令牌，用于后续接口调用。
3. 系统需要支持退出登录功能，退出后清除本地登录状态。
4. 系统需要支持 Token 刷新机制，避免用户在正常使用过程中频繁重新登录。
5. 系统需要提供当前登录用户信息接口，用于前端初始化用户信息、角色信息和菜单信息。
6. 登录失败、账号禁用、密码错误等情况需要返回明确提示。

# 用户角色和权限管理说明

1. 用户管理用于维护所有可登录系统的账号。
2. 角色管理用于维护系统中的权限分组。
3. 一个用户可以配置多个角色。
4. 一个角色可以配置多个用户。
5. 一个角色可以访问多个菜单。
6. 一个菜单可以被多个角色访问。
7. 用户登录后，系统根据用户角色获取可访问菜单。
8. 前端根据后端返回的菜单树动态生成侧边栏和页面路由。
9. 未授权菜单不能在前端展示，未授权接口需要在后端拦截。

# 组织人员基础数据说明

1. 系统需要提供部门维护功能，用于维护组织结构。
2. 系统需要提供岗位维护功能，用于维护人员职责分类。
3. 用户信息中需要关联部门和岗位。
4. 后续业务模块可以基于部门、岗位、角色进行数据范围控制。
5. 第一阶段可以只实现基础组织结构，不强制实现复杂的数据权限。

# 数据字典功能说明

1. 数据字典用于统一维护系统中的枚举类数据。
2. 字典需要分为字典类型和字典项。
3. 字典类型包括编码、名称、状态、说明等字段。
4. 字典项包括字典类型、字典值、字典标签、排序、状态等字段。
5. 业务表单中的下拉选择框可以从数据字典中读取选项。
6. 字典项停用后，不应继续出现在新增表单中，但历史数据仍然需要正常展示。

# 文件上传和附件管理说明

1. 系统需要提供统一的文件上传接口。
2. 文件上传后需要保存文件名称、文件地址、文件类型、文件大小、上传人、上传时间等元数据。
3. 业务模块可以通过附件 ID 或附件 URL 引用文件。
4. 文件类型需要做基础限制，仅允许上传常见办公文档和图片格式，禁止上传可执行文件、脚本文件等危险文件类型。
5. 图片类附件需要支持在页面中预览。
6. 文档类附件需要支持下载。
7. 文件访问需要基于用户登录态进行权限控制，未登录用户不能直接通过 URL 访问文件。

# 通知和消息中心说明

1. 系统需要提供站内消息功能。
2. 消息类型包括系统公告、待办提醒、处理结果通知等。
3. 消息需要记录接收人、消息标题、消息内容、消息类型、已读状态、发送时间等信息。
4. 用户登录后可以查看自己的消息列表，未读消息需要以醒目方式标识。
5. 用户可以将消息标记为已读，已读后不再显示未读标记。
6. 后续业务模块可以调用消息中心发送待办或结果通知。

# 操作日志和审计说明

1. 系统需要记录用户登录日志。
2. 系统需要记录关键操作日志，如新增、修改、删除、审核、导入、导出等。
3. 操作日志需要记录操作人、操作时间、操作模块、操作类型、请求参数、操作结果等信息。
4. 系统需要记录接口访问异常和系统错误日志。
5. 日志信息主要用于问题排查、合规审计和操作追踪。

# 系统配置功能说明

1. 系统需要提供基础参数配置能力。
2. 配置项包括系统名称、登录页标题、文件上传限制、Token 有效期等。
3. 配置项需要支持启用和停用。
4. 配置变更需要记录操作日志。
5. 第一阶段可以先实现数据库配置，不强制实现复杂的动态热更新。

# 基础工作台说明

1. 系统首页需要提供基础工作台。
2. 工作台需要显示当前用户待办数量、消息数量、最近操作记录、系统公告等信息。
3. 管理员可以看到系统用户数、角色数、菜单数、今日登录数、接口异常数等统计信息。
4. 首页布局包括顶部栏、左侧菜单栏、中间功能展示区域。
5. 点击左侧菜单后，中间区域展示对应功能页面。

# 系统开发要求

## 项目架构

采用 pnpm workspace Monorepo 结构，分为 `frontend` 和 `backend` 两个子包：

- **frontend**：前端 SPA 应用，技术栈为 **Vue 3 + Vite 7 + TypeScript + Element Plus + Tailwind CSS 4**
- **backend**：后端 API 服务，技术栈为 **Fastify 5 + TypeScript + MySQL2 + JWT**

## 文档要求

1. 需要生成独立的需求文档，每个一级功能模块生成一个需求文档，Markdown 格式，存放在 `SPECS` 目录下。
2. 需要生成完整的数据库设计文档，Markdown 格式，存放在 `SPECS` 目录下，包含用户、角色、菜单、部门、岗位、字典、附件、消息、日志、配置等核心表。
3. 需要生成前后端接口设计文档，Markdown 格式，明确接口地址、请求参数、返回字段和权限要求，存放在 `SPECS` 目录下。

## 前端要求

4. 基于 **Vue 3 Composition API** + `<script setup>` 语法编写组件。
5. 使用 **Element Plus** 作为 UI 组件库，实现表单、表格、弹窗、菜单等交互组件。
6. 使用 **Tailwind CSS 4** 进行布局和样式微调，与 Element Plus 配合使用。
7. 使用 **Vue Router 4** 实现前端路由，支持动态路由（根据用户角色从后端获取菜单后动态注册）。
8. 使用 **Pinia** 进行全局状态管理，管理用户信息、角色、菜单、Token 等状态。
9. 使用 **Axios** 封装 HTTP 请求，统一处理请求拦截（注入 Token）、响应拦截（Token 过期刷新、错误提示）。
10. 前端开发端口为 `8848`，通过 Vite proxy 将 `/api` 请求代理到后端 `http://localhost:3000`。

## 后端要求

11. 基于 **Fastify 5** 构建 RESTful API 服务，使用 TypeScript 编写。
12. 使用 **JWT（jsonwebtoken）** 实现 Token 认证，登录成功返回 Access Token 和 Refresh Token。
13. 使用 **MySQL2** 连接 MySQL 数据库，数据库名称为 `admin_template`。
14. 使用 **Winston** 记录应用日志（操作日志、错误日志、访问日志）。
15. 后端服务端口为 `3000`。

## Demo 交付要求

16. Demo 需要包含登录、动态菜单、用户管理、角色管理、菜单管理、字典管理、文件上传、消息中心、日志查看、系统配置、个人信息等基础页面。
17. Demo 页面需要可访问、可交互，所有 CRUD 操作可正常执行。
18. 后端 API 需要提供完整的请求校验、错误处理和统一响应格式 `{ code, data, message }`。
19. 前端与后端需要真实联调，通过 Axios 调用后端 API，不使用前端 Mock 数据。
20. Demo 启动方式：根目录执行 `pnpm dev` 启动前后端，前端访问 `http://localhost:8848`，后端 API 访问 `http://localhost:3000`。
